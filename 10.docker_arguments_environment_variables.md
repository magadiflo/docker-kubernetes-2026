# ⚙️ Sección 10: Docker - Arguments y Environment Variables

---

## 🌐 [Introducción: El Ciclo de Vida de las Variables](https://vsupalov.com/docker-arg-env-variable-guide/)

En `Docker`, la gestión de configuraciones se divide en dos mundos: el tiempo de construcción `(Build-time)`
y el tiempo de ejecución `(Run-time)`. Comprender la diferencia entre `ARG` y `ENV` es vital para crear imágenes
portables y seguras.

### 🗺️ Mapa de Disponibilidad

No todas las variables viven para siempre. El siguiente esquema resume cuándo nace
y dónde muere cada tipo de variable  `ARG` y `ENV`:

![01.png](assets/section-10/01.png)

### 🏗️ ARG: Build-time Arguments

Las variables definidas mediante `ARG` son **efímeras**. Su propósito es inyectar valores durante el proceso de
empaquetado de la imagen.

- `Alcance`: Solo existen dentro del `Dockerfile`.
- `Limitación`: Los procesos internos del contenedor (como tu aplicación Spring Boot o scripts en CMD) no tienen acceso
  a estos valores.
- `Interactividad`: Si defines un `ARG` sin valor por defecto y no le proporcionas uno mediante `--build-arg`,
  **Docker no lanzará un error automáticamente** — el argumento simplemente tendrá un valor vacío.
  El fallo ocurrirá solo si algún paso del `Dockerfile` depende de ese valor para ejecutarse correctamente,
  como un `FROM` o un `RUN`.

````bash
# Si no se pasa --build-arg APP_VERSION, el argumento quedará vacío y podría
# causar fallos en pasos posteriores, solo si algún paso depende de este valor
ARG APP_VERSION
````

### 🚀 ENV (Build-time and run-time Environment Variables)

Las variables `ENV` están disponibles tanto durante la construcción como para el futuro contenedor en ejecución.
**En el `Dockerfile` son utilizables tan pronto como se introducen con una instrucción `ENV`.**

A diferencia de `ARG`, los valores `ENV` persisten en la capa de metadatos de la imagen final y son plenamente
accesibles por el runtime del contenedor. Además, Docker permite la `inyección dinámica`, lo que significa que
cualquier valor `ENV` definido en el `Dockerfile` puede ser sobrescrito en tiempo de ejecución mediante las
banderas `-e` o `--env`.

Otra distinción importante es que `ENV` **siempre requiere un valor en su declaración dentro del Dockerfile**. No es
posible declarar una variable `ENV` sin asignarle un valor inicial, aunque este puede ser sobreescrito posteriormente
en tiempo de ejecución.

### 📊 Comparativa Técnica: ARG vs. ENV

> Docker soporta dos tipos de variables que podemos configurar:
> - `Arguments` en tiempo de construcción de la imagen
> - `Environment` en tiempo de ejecución.

| Característica                     | **ARG** (Arguments)                                 | **ENV** (Environment)                                                                                                                         |
|------------------------------------|-----------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------|
| **Uso Principal**                  | Configurar la construcción de la imagen.            | Configurar la ejecución de la aplicación.                                                                                                     |
| **Visibilidad en Código**          | ❌ No accesible por la App.                          | ✅ Accesible (ej. `System.getenv()`).                                                                                                          |
| **Instrucciones `CMD/ENTRYPOINT`** | ❌ No disponible.                                    | ✅ Disponible.                                                                                                                                 |
| **Persistencia**                   | ❌ Se pierde tras el `build`.                        | ✅ Persiste en el contenedor.                                                                                                                  |
| **Comando de Inyección**           | `docker build --build-arg ...`                      | `docker run -e ...`                                                                                                                           |
| **Declaración sin valor**          | ✅ Válido — queda vacío si no se pasa `--build-arg`. | ❌ No válido — siempre requiere un valor inicial en el `Dockerfile`. Ese valor inicial puede ser sobreescrito en tiempo de ejecución con `-e`. |

> `Regla de Oro`:
> - Usa `ARG` para versiones de librerías, nombres de archivos temporales o configuraciones de compilación.
> - Usa `ENV` para credenciales de BD, URLs de servicios o parámetros que cambian entre Desarrollo y Producción.

## 🚀 Uso de Variables de Entorno `(ENV)` en `user-service`

En esta etapa, transformaremos el microservicio `user-service` en un componente dinámico. El objetivo es asignar
dinámicamente un **puerto de escucha** y el **perfil de ejecución** sin necesidad de reconstruir la imagen para cada
cambio.

## 🐳 A) Variables de Entorno `(ENV)` a nivel de Dockerfile

### 1. Preparación en Spring Boot (`application.yml`)

Lo primero que haremos será modificar el `application.yml` del `user-service` donde agregaremos la variable de
entorno `CONTAINER_PORT`.

````yml
server:
  port: ${CONTAINER_PORT:8001}
````

- `CONTAINER_PORT`: Nombre de la variable que buscaremos en el entorno de Docker.
- `8001`: Valor por defecto. Si la variable de entorno `CONTAINER_PORT` no fue definida, la app arrancará en el puerto
  `8001`.

### 2. Definición en el `Dockerfile`

En el `Dockerfile` del `user-service` utilizamos la instrucción `ENV` para establecer un valor predeterminado a nivel
de imagen. Esto asegura que la variable siempre exista.

````Dockerfile
# ... instrucciones anteriores ...

# Establecemos el puerto 8000 como valor por defecto de la imagen
ENV CONTAINER_PORT=8000

# Informamos que el contenedor escuchará en el puerto definido
EXPOSE ${CONTAINER_PORT}

ENTRYPOINT ["java", "org.springframework.boot.loader.launch.JarLauncher"]
````

- `ENV`, esta instrucción establece la variable de entorno `CONTAINER_PORT` con valor de puerto `8000`. Este valor
  estará en el entorno para todas las instrucciones posteriores en la etapa de construcción y puede ser reemplazado en
  línea.
- Al definir una variable de entorno con `ENV` `sí o sí debemos asignarle un valor`. La instrucción `ENV` requiere
  obligatoriamente la asignación de un valor inicial en el Dockerfile. A diferencia de `ARG`, no puede declararse
  'vacía'; esto asegura que la imagen tenga un estado predecible desde su construcción.
- Es importante recordar que, en esta situación, aunque `ENV` puede ser utilizado tanto en la construcción como en la
  ejecución del contenedor, cuando se usa en `EXPOSE`, su efecto es únicamente durante la fase de construcción de la
  imagen. Así que en este caso, el `ENV` está funcionando solo para la configuración del puerto en la imagen
  construida.

**💡Nota**
> Recordemos que en el `application.yml` del `user-service` hemos definido como valor por defecto el `8001` si es que
> la variable de entorno `CONTAINER_PORT` no viene definida.
>
> Ahora, como en el `Dockerfile` estamos estableciendo explícitamente el `ENV CONTAINER_PORT=8000`, significa que
> esa variable siempre estará definida y con valor `8000`, por lo que el puerto por defecto `(8001)` del
> `application.yml` jamás se usará.

### 4. Ejecución del Contenedor

Al ejecutar, debemos ser coherentes con el mapeo de puertos `(-p)`. Si nuestra imagen define internamente el
puerto `8000`, el mapeo debe apuntar a ese valor:

````bash
$ docker container run -d -p 8001:8000 --rm --name c-user-service --network docker-k8s-net user-service
c1dbac18aaf7d25894abd04bcdad816da26b2fd433e7e9dda2553f10c4a97b6a
````

- `-p 8001:8000`, el puerto externo sigue siendo `8001`, recordemos que usamos ese valor para poder acceder desde
  nuestra máquina local al contendor. Por otro lado, el cambio que hemos realizado fue en el puerto interno `8000`.
  Esto significa que al interior del contenedor nuestra aplicación de Spring Boot usará dicho puerto. Para que eso
  suceda, el valor del puerto interno definido en el parámetro `-p 8001:8000` debe ser coherente con el valor que le
  definimos a la variable `ENV CONTAINER_PORT=8000` dentro del `Dockerfile`.

## 🐳 B) Sobrescritura de variables `ENV` por línea de comandos

La verdadera potencia de Docker reside en su capacidad de inyectar configuraciones en tiempo de ejecución. Esto nos
permite usar la `misma imagen` para diferentes entornos (`dev`, `test`, `prod`) simplemente variando los parámetros
de inicio.

### 1. Preparación de Perfiles en Spring Boot

En el `application.yml`, configuramos Spring para que busque un perfil activo (busque la variable de entorno
`SPRING_PROFILES_ACTIVE`). Si no lo encuentra, usará `default`.

````yml
server:
  port: ${CONTAINER_PORT:8001}
  error:
    include-message: always

spring:
  application:
    name: user-service
  profiles:
    active: ${SPRING_PROFILES_ACTIVE:default}
````

> `Diferencia de Estrategia`: A diferencia de `CONTAINER_PORT`, **no definiremos** `SPRING_PROFILES_ACTIVE`
> en el `Dockerfile`. Esto nos permite probar cómo Docker inyecta variables que la aplicación espera,
> aunque no existan en la metadata de la imagen.
>
> La variable `SPRING_PROFILES_ACTIVE` la definiremos al momento de crear un contenedor del `user-service`.

### 5. Ejecución Dinámica con la bandera `-e`

Utilizaremos las banderas `-e` (o `--env`) para forzar nuevos valores y sobrescribir los predeterminados. Es decir,
crearemos un contenedor asignando a través de la línea de comandos:

1. La variable de entorno que tenemos definido en el `Dockerfile`: `CONTAINER_PORT`.
2. La variable de entorno que solo está definida en nuestra aplicación: `SPRING_PROFILES_ACTIVE`.

De esta manera, sobreescribiremos los valores por defecto que tienen definidos.

````bash
$ docker container run -d -p 8001:8090 --rm --name c-user-service -e SPRING_PROFILES_ACTIVE=dev -e CONTAINER_PORT=8090 --network docker-k8s-net user-service
05304f45b53503091141b77829bcfcaaac2056f033f5a6b27421372db84706ba
````

**Donde**

- `-p 8001:8090`, cambiamos el puerto interno a `8090` para que coincida con nuestra variable inyectada.
- `-e` o `--env`, nos permite definir una variable de entorno.
- `CONTAINER_PORT=8090`, variable de entorno definida en la línea de comandos. Esta variable sobreescribe a la variable
  que definimos en el `Dockerfile`, si es que en ese archivo existe dicha variable. Caso contrario, simplemente estamos
  creando la variable para que sea usada por quien la defina al interior del contenedor. En nuestro caso,
  el `application.yml` en la configuración `server.port`.
- `SPRING_PROFILES_ACTIVE=dev`, variable de entorno que no ha sido definido en el `Dockerfile`, pero que sí está
  definido en el `application.yml` de nuestro `user-service`.

**Nota**
> Las variables de entorno que definamos a través de la línea de comandos, no necesariamente tiene que estar definida
> en el `Dockerfile`, es decir, a través de la línea de comandos podemos definir variables de entorno, y si existen
> en el `Dockerfile`, obviamente se sobreescribirán, en caso de que no existan, simplemente estarán disponibles en el

## 🚀 Trabajando con argumentos `(ARG)` en el Dockerfile del `user-service`

Antes de empezar a trabajar con argumentos `(ARG)`, considero importante revisar la siguiente teoría:

### [🌐 Entendiendo cómo interactúan ARG y FROM](https://docs.docker.com/engine/reference/builder/#understand-how-arg-and-from-interact)

Las instrucciones `FROM` soportan variables que son declaradas por cualquier instrucción `ARG` que ocurra antes del
primer `FROM`. Es decir, podemos declarar al inicio de todas las etapas variables del tipo `ARG` y la instrucción
`FROM` de cada etapa las podrá usar sin problema.

**Veamos el siguiente ejemplo (no es parte del proyecto):**

````dockerfile
# Definición GLOBAL: Disponible para todos los FROM
ARG  CODE_VERSION=latest

FROM base:${CODE_VERSION}
CMD  /code/run-app

FROM extras:${CODE_VERSION}
CMD  /code/run-extras
````

### 🧱 Uso de variable global `ARG` dentro de una etapa de construcción `(stage)`

Cuando defines un `ARG` antes de cualquier instrucción `FROM`, ese argumento es considerado global y puede usarse en la
instrucción `FROM` inmediatamente posterior. Sin embargo, para que esté disponible dentro de una etapa de construcción
(por ejemplo, para usarlo en un `RUN`, `ENV`, o cualquier otra instrucción dentro del bloque que sigue al `FROM`),
es necesario volver a declarar el mismo `ARG` dentro de esa etapa.

> En resumen: los `ARG` globales pueden usarse en el `FROM`, pero si quieres acceder a su valor dentro del contenido
> de la etapa, debes declarar nuevamente el `ARG` (sin necesidad de asignarle un valor).

````dockerfile
# 1. Definición Global
ARG VERSION=latest

# --- Inicio de Etapa ---
FROM busybox:${VERSION}
# 2. Re-declaración necesaria para "bajar" el valor a la etapa
ARG VERSION
# 3. Ahora sí es accesible para procesos internos
RUN echo ${VERSION} > image_version
# --- Fin de Etapa ---
````

En el ejemplo anterior, `ARG VERSION` se declara primero de forma global, lo que permite usarlo en el `FROM`.
Luego se vuelve a declarar dentro de la etapa para que su valor esté disponible en las instrucciones siguientes,
como el `RUN`.

### Resumen de Visibilidad:

- `Antes del FROM`: Alcance Global (Solo para otros `FROM`).
- `Después del FROM`: Alcance de Etapa (Solo para `RUN`, `CMD`, `COPY`, etc., dentro de ese bloque).
- `Re-declaración`: Es el puente que permite que una etapa "vea" un valor definido globalmente.

## 🏗️ Implementación de `ARG` en el `Dockerfile`

Para hacer nuestra imagen más flexible, hemos parametrizado las versiones de las imágenes base de Eclipse Temurin.
Esto nos permite actualizar la versión de Java (JDK/JRE) sin necesidad de editar el contenido del `Dockerfile`.

### 1. Configuración del Dockerfile Parametrizado

Declaramos los argumentos al inicio `(Scope Global)` para que todas las etapas (`dependencies`, `builder`, `runner`)
puedan heredar estos valores.

````dockerfile
# Definición de argumentos globales con valores por defecto
ARG JDK_VERSION=25-jdk-alpine
ARG JRE_VERSION=25-jre-alpine

# Etapa 1: Construcción (Usa JDK)
FROM eclipse-temurin:${JDK_VERSION} AS dependencies
WORKDIR /app
COPY ./mvnw ./
COPY ./.mvn ./.mvn
COPY ./pom.xml ./
RUN sed -i -e 's/\r$//' ./mvnw && ./mvnw dependency:go-offline
COPY ./src ./src
RUN ./mvnw clean package -DskipTests

# Etapa 2: Extracción (Usa JRE)
FROM eclipse-temurin:${JRE_VERSION} AS builder
WORKDIR /app
COPY --from=dependencies /app/target/*.jar ./app.jar
RUN java -Djarmode=layertools -jar app.jar extract

# Etapa 3: Ejecución Final (Usa JRE)
FROM eclipse-temurin:${JRE_VERSION} AS runner
WORKDIR /app
RUN mkdir ./logs
COPY --from=builder /app/dependencies ./
COPY --from=builder /app/spring-boot-loader ./
COPY --from=builder /app/snapshot-dependencies ./
COPY --from=builder /app/application ./
ENV CONTAINER_PORT=8001
EXPOSE ${CONTAINER_PORT}
ENTRYPOINT ["java", "org.springframework.boot.loader.launch.JarLauncher"]
````

#### 🔍 Análisis Técnico de `ARG`

- `Flexibilidad Dinámica`: Gracias a esta configuración, si el día de mañana queremos cambiar a `Java 17`,
  no editamos el archivo. Simplemente ejecutamos:
  ````bash
  $ docker image build --build-arg JDK_VERSION=17-jdk-alpine --build-arg JRE_VERSION=17-jre-alpine -t user-service .
  ````
  **El punto final `(.)` en el comando de construcción:** Representa el `Contexto de Construcción (Build Context)`.
  Es el directorio que Docker "mapea" y envía al motor de Docker para que instrucciones como COPY o ADD puedan
  encontrar los archivos de tu proyecto (como el `pom.xml` o la carpeta `src`). Típicamente, es el lugar donde reside
  el `Dockerfile`.


- `Valores por Defecto`: Si no se utiliza la bandera `--build-arg`, Docker usará automáticamente las versiones `25` que
  definimos como fallback.
- `Ciclo de Vida`: Es vital recordar que `JDK_VERSION` y `JRE_VERSION` dejan de existir una vez finalizada
  la construcción. No ocupan espacio en los metadatos del contenedor en ejecución.

## ⚡ Sobrescritura de `ARG` por línea de comandos

Al igual que con las variables de entorno, Docker nos permite ignorar los valores por defecto de un `Dockerfile`
y pasar nuevos argumentos durante la fase de construcción. Esto se logra mediante la bandera `--build-arg`.

### 1. Construcción con argumentos personalizados

En este ejemplo, forzaremos el uso de una versión específica de `Alpine (3.23)` para nuestras imágenes de `JDK`

### 💡 Resumen de Diferencias: `ARG` vs `ENV`

| Característica     | ARG (Build Argument)                         | ENV (Environment Variable)                     |
|--------------------|----------------------------------------------|------------------------------------------------|
| **Momento de uso** | Durante el `docker build`.                   | Durante el `docker run`.                       |
| **Persistencia**   | **No** persiste en la imagen final.          | **Sí** persiste y es visible en el contenedor. |
| **Uso principal**  | Versiones de software, rutas de compilación. | Credenciales, puertos, perfiles de Spring.     |
| **Comando**        | `--build-arg`                                | `-e` o `--env-file`                            |

## 🗄️ user-service: Externalización de la configuración de MySQL (`ENV`)

Un microservicio profesional nunca debe tener URLs o credenciales de base de datos hardcodeadas. Utilizaremos el
mecanismo de `Spring Boot Property Placeholder` para inyectar estos valores desde Docker.

### 1. Configuración dinámica en `application.yml` del `user-service`

Modificamos el bloque de `datasource` para que Spring busque variables de entorno antes de arrancar. Si las variables
no existen, el despliegue fallará, lo cual es deseable para evitar conexiones erróneas.

````yml
spring:
  datasource:
    url: jdbc:mysql://${DB_HOST}:${DB_PORT}/${DB_NAME}
    username: ${DB_USERNAME}
    password: ${DB_PASSWORD}
````
