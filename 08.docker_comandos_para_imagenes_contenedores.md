## 🖼️ Comandos para imágenes

### 🐳 Listar imágenes

Para visualizar todas las imágenes disponibles en el host local, incluyendo aquellas que son capas intermedias:

````bash
$ docker image ls -a                                                                      
                                                                                          
IMAGE                                    ID             DISK USAGE   CONTENT SIZE   EXTRA    
user-service:latest                      65cf101a09ed        294MB             0B    U    
````

### 🐳 Eliminación de Imágenes `(rm)`

La eliminación de imágenes requiere que no existan contenedores (activos o detenidos) asociados a ellas. **Si la imagen
tiene contenedores ejecutándose mostrará mensajes de error y no podrá eliminarse.**

- `Por nombre (Default Tag)`: Si no se especifica el tag, Docker intentará eliminar `latest`.
  ````bash
  $ docker image rm user-service
  Untagged: user-service:latest
  Deleted: sha256:65cf101a09ed0a94248a09b3a086075a24f9dd25c067873398dc1c9240b3a6e8
  ````
- `Por Tag específico`: Útil cuando manejamos versiones (`v1`, `v2`, etc.).
  ````bash
  $ docker image ls -a                                                                     
                                                                                           
  IMAGE                                    ID             DISK USAGE   CONTENT SIZE   EXTRA       
  user-service:latest                      65cf101a09ed        294MB             0B        
  user-service:v1                          65cf101a09ed        294MB             0B        
  ````
  Ahora sí, eliminamos la imagen `user-service` con tag `v1`.
  ````bash
  $ docker image rm user-service:v1
  Untagged: user-service:v1
  ````

### 🐳 Limpieza Automática (`prune`)

Con el tiempo, las construcciones fallidas o actualizaciones dejan imágenes "huérfanas" (dangling images) que no tienen
etiqueta y consumen espacio. Para realizar esa limpieza podemos usar el comando `docker image prune`. A continuación
veremos un ejemplo.

Al listar todas las imágenes, es común encontrar entradas marcadas como `<untagged>` o `<none>`.
Estas son imágenes que han perdido su relación con un `nombre` y `tag` (usualmente porque fueron reemplazadas por
una versión más nueva con el mismo nombre).

````bash
$ docker image ls -a                                                                     
                                                                                         
IMAGE                                    ID             DISK USAGE   CONTENT SIZE   EXTRA        
redis:8.0.3-alpine                       c25e2f66b829       60.5MB             0B        
sonarqube:25.10.0.114319-community       047bd8988268       1.23GB             0B        
testcontainers/ryuk:0.12.0               a926383422af       15.8MB             0B        
user-service:latest                      65cf101a09ed        294MB             0B        
<untagged>                               f51ac432965a        294MB             0B                 
````

Estas imágenes ocupan espacio en disco innecesariamente y pueden ser eliminadas de forma segura con el comando
`docker image prune`.

````bash
$ docker image prune                                                            
WARNING! This will remove all dangling images.                                  
Are you sure you want to continue? [y/N] y                                      
Deleted Images:                                                                 
deleted: sha256:f51ac432965ae2b8287eca992808655364699d9460a191a8fa576720366519fd
                                                                                
Total reclaimed space: 0B                                                       
````

Tras la limpieza, si volvemos a listar las imágenes, notaremos que el entorno está depurado y solo se conservan
las imágenes con nombre y etiqueta (TAG) asignada:

````bash
$ docker image ls -a                                                                     
                                                                                         
IMAGE                                    ID             DISK USAGE   CONTENT SIZE   EXTRA       
redis:8.0.3-alpine                       c25e2f66b829       60.5MB             0B        
sonarqube:25.10.0.114319-community       047bd8988268       1.23GB             0B        
testcontainers/ryuk:0.12.0               a926383422af       15.8MB             0B        
user-service:latest                      65cf101a09ed        294MB             0B                 
````

### 🐳 Auditoría Profunda: `docker image inspect`

El comando `inspect` devuelve un objeto JSON con los metadatos completos de la imagen. Es la herramienta principal para
verificar la configuración interna sin tener que arrancar el contenedor.

| Campo          | Descripción Técnica                                                           |
|----------------|-------------------------------------------------------------------------------|
| `Id`           | Hash SHA256 único que identifica el contenido exacto de la imagen.            |
| `ExposedPorts` | Puertos definidos en el contrato del `Dockerfile`.                            |
| `Env`          | Variables de entorno heredadas (ej. `JAVA_VERSION`).                          |
| `Entrypoint`   | El binario y argumentos que inician la aplicación.                            |
| `Layers`       | La pila de sistemas de archivos. Vital para validar la optimización de caché. |
| `Size`         | Peso total en disco (en bytes).                                               |

### 🐳 Etiquetado de Imágenes (Versioning)

El etiquetado o tagging es la forma en la que gestionamos las versiones de nuestros microservicios.
Si no se especifica una etiqueta durante la construcción, Docker asigna automáticamente el valor `latest`.

Hasta el momento ya tenemos creado la imagen `user-service` con la etiqueta `latest`, que es la etiqueta por defecto.

````bash
$ docker image ls -a                                                                     
                                                                                         
IMAGE                                    ID             DISK USAGE   CONTENT SIZE   EXTRA
user-service:latest                      65cf101a09ed        294MB             0B         
````

#### 🏷️ Concepto Clave: Un ID, Múltiples Etiquetas

Al listar las imágenes, observamos algo interesante:

````bash
$ λ docker image ls -a                                                                     
                                                                                         
IMAGE                                    ID             DISK USAGE   CONTENT SIZE   EXTRA       
user-service:latest                      65cf101a09ed        294MB             0B        
user-service:v1                          65cf101a09ed        294MB             0B        
````

💡 **¿Qué significa esto?**

- En Docker, el ID es el identificador real del contenido (el hash de las capas). Las etiquetas (`latest`, `v1`) son
  simplemente punteros o "alias" que apuntan a ese ID.
- Importante: Aunque veas dos entradas de `294MB`, en tu disco duro solo se ocupan `294MB` en total, no el doble. Ambas
  etiquetas comparten los mismos archivos físicos.

### 🐳 Renombrado y Alias: El comando `docker image tag`

El comando `docker image tag` no crea una nueva imagen ni modifica los archivos internos; simplemente crea un
alias o una nueva referencia que apunta al mismo `IMAGE ID`.

Esta operación es instantánea y es el paso previo esencial antes de subir una imagen a un registro remoto como Docker
Hub.

#### 🏷️ Creación de un Alias para Repositorio Remoto

Para preparar nuestra imagen para ser subida a una cuenta de `Docker Hub`, solemos seguir el patrón:
`usuario/nombre-imagen:tag`.

````bash
$ docker image tag user-service:latest magadiflo/user-service:v2
````

Después de ejecutar el comando anterior, listamos las imágenes para comprobar la existencia de la nueva etiqueta.

````bash
$ docker image ls -a                                                                     
                                                                                         
IMAGE                                    ID             DISK USAGE   CONTENT SIZE   EXTRA
magadiflo/user-service:v2                65cf101a09ed        294MB             0B        
user-service:latest                      65cf101a09ed        294MB             0B        
user-service:v1                          65cf101a09ed        294MB             0B        
````

1. `Identidad Única`: Fíjate que el ID `65cf101a09ed` se repite en las tres filas. Esto confirma que Docker no ha
   duplicado los datos en tu disco duro; solo ha registrado tres nombres diferentes para el mismo contenido.
2. `Nomenclatura de Distribución`: Al agregar el prefijo `magadiflo/`, Docker ahora sabe que esta imagen pertenece a ese
   espacio de nombres (namespace). Si ejecutaras un docker push, Docker intentaría subirla automáticamente a tu
   repositorio en la nube.
3. `Independencia de Eliminación`: Si eliminas la etiqueta `user-service:v1`, la imagen física no se borrará mientras
   existan otras etiquetas (como `latest` o `v2`) apuntando al mismo ID.

#### 🔍 Desglose de componentes:

- `docker container run`: Es el comando principal que se encarga de crear una nueva instancia (contenedor) a partir de
  una imagen y ponerla en marcha.
- `-p 8085:8001`: Establece el mapeo de puertos en formato `Host:Contenedor`. El tráfico dirigido al puerto 8085 de tu
  máquina física será redirigido al puerto 8001 interno del contenedor.
- `user-service`: Define la imagen base. Al no especificar un tag (etiqueta), Docker buscará automáticamente la versión
  `latest`.
- `¿Qué es el modo attach?`: Es el comportamiento por defecto de Docker donde los flujos de entrada, salida y error
  estándar (`stdin`, `stdout`, `stderr`) del contenedor se vinculan a tu terminal actual.

> 💡 `Nota de control`: En este modo, si cierras la terminal o presionas `Ctrl + C`, enviarás una señal de interrupción
> al proceso principal de Spring Boot, lo que causará que el contenedor se detenga.

### 📦 Des-atachar deteniendo el contenedor

Si queremos volver a tener el control de la línea de comando debemos presionar `Ctrl + C`, pero **esto detendrá el
contenedor**, cambiando su status `Up` por `Exited`.

````bash
$ docker container ls -a

````
