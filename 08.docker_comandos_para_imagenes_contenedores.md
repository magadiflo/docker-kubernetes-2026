## 🖼️ Comandos para imágenes

### 🐳 Listar imágenes

Para visualizar todas las imágenes disponibles en el host local, incluyendo aquellas que son capas intermedias:

````bash
$ docker image ls -a                                                                      
                                                                                          
IMAGE                                    ID             DISK USAGE   CONTENT SIZE   EXTRA    
user-service:latest                      65cf101a09ed        294MB             0B    U    
````

### 🐳 Eliminación de Imágenes `(rm)`

La eliminación de imágenes requiere que no existan contenedores (activos o detenidos) asociados a ellas. **Si la imagen
tiene contenedores ejecutándose mostrará mensajes de error y no podrá eliminarse.**

- `Por nombre (Default Tag)`: Si no se especifica el tag, Docker intentará eliminar `latest`.
  ````bash
  $ docker image rm user-service
  Untagged: user-service:latest
  Deleted: sha256:65cf101a09ed0a94248a09b3a086075a24f9dd25c067873398dc1c9240b3a6e8
  ````
- `Por Tag específico`: Útil cuando manejamos versiones (`v1`, `v2`, etc.).
  ````bash
  $ docker image ls -a                                                                     
                                                                                           
  IMAGE                                    ID             DISK USAGE   CONTENT SIZE   EXTRA       
  user-service:latest                      65cf101a09ed        294MB             0B        
  user-service:v1                          65cf101a09ed        294MB             0B        
  ````
  Ahora sí, eliminamos la imagen `user-service` con tag `v1`.
  ````bash
  $ docker image rm user-service:v1
  Untagged: user-service:v1
  ````

### 🐳 Limpieza Automática (`prune`)

Con el tiempo, las construcciones fallidas o actualizaciones dejan imágenes "huérfanas" (dangling images) que no tienen
etiqueta y consumen espacio. Para realizar esa limpieza podemos usar el comando `docker image prune`. A continuación
veremos un ejemplo.

Al listar todas las imágenes, es común encontrar entradas marcadas como `<untagged>` o `<none>`.
Estas son imágenes que han perdido su relación con un `nombre` y `tag` (usualmente porque fueron reemplazadas por
una versión más nueva con el mismo nombre).

````bash
$ docker image ls -a                                                                     
                                                                                         
IMAGE                                    ID             DISK USAGE   CONTENT SIZE   EXTRA        
redis:8.0.3-alpine                       c25e2f66b829       60.5MB             0B        
sonarqube:25.10.0.114319-community       047bd8988268       1.23GB             0B        
testcontainers/ryuk:0.12.0               a926383422af       15.8MB             0B        
user-service:latest                      65cf101a09ed        294MB             0B        
<untagged>                               f51ac432965a        294MB             0B                 
````

Estas imágenes ocupan espacio en disco innecesariamente y pueden ser eliminadas de forma segura con el comando
`docker image prune`.

````bash
$ docker image prune                                                            
WARNING! This will remove all dangling images.                                  
Are you sure you want to continue? [y/N] y                                      
Deleted Images:                                                                 
deleted: sha256:f51ac432965ae2b8287eca992808655364699d9460a191a8fa576720366519fd
                                                                                
Total reclaimed space: 0B                                                       
````

Tras la limpieza, si volvemos a listar las imágenes, notaremos que el entorno está depurado y solo se conservan
las imágenes con nombre y etiqueta (TAG) asignada:

````bash
$ docker image ls -a                                                                     
                                                                                         
IMAGE                                    ID             DISK USAGE   CONTENT SIZE   EXTRA       
redis:8.0.3-alpine                       c25e2f66b829       60.5MB             0B        
sonarqube:25.10.0.114319-community       047bd8988268       1.23GB             0B        
testcontainers/ryuk:0.12.0               a926383422af       15.8MB             0B        
user-service:latest                      65cf101a09ed        294MB             0B                 
````

### 🐳 Etiquetado de Imágenes (Versioning)

El etiquetado o tagging es la forma en la que gestionamos las versiones de nuestros microservicios.
Si no se especifica una etiqueta durante la construcción, Docker asigna automáticamente el valor `latest`.

Hasta el momento ya tenemos creado la imagen `user-service` con la etiqueta `latest`, que es la etiqueta por defecto.

````bash
$ docker image ls -a                                                                     
                                                                                         
IMAGE                                    ID             DISK USAGE   CONTENT SIZE   EXTRA
user-service:latest                      65cf101a09ed        294MB             0B         
````

#### 🏷️ Concepto Clave: Un ID, Múltiples Etiquetas

Al listar las imágenes, observamos algo interesante:

````bash
$ λ docker image ls -a                                                                     
                                                                                         
IMAGE                                    ID             DISK USAGE   CONTENT SIZE   EXTRA       
user-service:latest                      65cf101a09ed        294MB             0B        
user-service:v1                          65cf101a09ed        294MB             0B        
````

💡 **¿Qué significa esto?**

- En Docker, el ID es el identificador real del contenido (el hash de las capas). Las etiquetas (`latest`, `v1`) son
  simplemente punteros o "alias" que apuntan a ese ID.
- Importante: Aunque veas dos entradas de `294MB`, en tu disco duro solo se ocupan `294MB` en total, no el doble. Ambas
  etiquetas comparten los mismos archivos físicos.

### 🐳 Renombrado y Alias: El comando `docker image tag`

El comando `docker image tag` no crea una nueva imagen ni modifica los archivos internos; simplemente crea un
alias o una nueva referencia que apunta al mismo `IMAGE ID`.

Esta operación es instantánea y es el paso previo esencial antes de subir una imagen a un registro remoto como Docker
Hub.

#### 🏷️ Creación de un Alias para Repositorio Remoto

Para preparar nuestra imagen para ser subida a una cuenta de `Docker Hub`, solemos seguir el patrón:
`usuario/nombre-imagen:tag`.

````bash
$ docker image tag user-service:latest magadiflo/user-service:v2
````

Después de ejecutar el comando anterior, listamos las imágenes para comprobar la existencia de la nueva etiqueta.

````bash
$ docker image ls -a                                                                     
                                                                                         
IMAGE                                    ID             DISK USAGE   CONTENT SIZE   EXTRA
magadiflo/user-service:v2                65cf101a09ed        294MB             0B        
user-service:latest                      65cf101a09ed        294MB             0B        
user-service:v1                          65cf101a09ed        294MB             0B        
````

1. `Identidad Única`: Fíjate que el ID `65cf101a09ed` se repite en las tres filas. Esto confirma que Docker no ha
   duplicado los datos en tu disco duro; solo ha registrado tres nombres diferentes para el mismo contenido.
2. `Nomenclatura de Distribución`: Al agregar el prefijo `magadiflo/`, Docker ahora sabe que esta imagen pertenece a ese
   espacio de nombres (namespace). Si ejecutaras un docker push, Docker intentaría subirla automáticamente a tu
   repositorio en la nube.
3. `Independencia de Eliminación`: Si eliminas la etiqueta `user-service:v1`, la imagen física no se borrará mientras
   existan otras etiquetas (como `latest` o `v2`) apuntando al mismo ID.

#### 🔍 Desglose de componentes:

- `docker container run`: Es el comando principal que se encarga de crear una nueva instancia (contenedor) a partir de
  una imagen y ponerla en marcha.
- `-p 8085:8001`: Establece el mapeo de puertos en formato `Host:Contenedor`. El tráfico dirigido al puerto 8085 de tu
  máquina física será redirigido al puerto 8001 interno del contenedor.
- `user-service`: Define la imagen base. Al no especificar un tag (etiqueta), Docker buscará automáticamente la versión
  `latest`.
- `¿Qué es el modo attach?`: Es el comportamiento por defecto de Docker donde los flujos de entrada, salida y error
  estándar (`stdin`, `stdout`, `stderr`) del contenedor se vinculan a tu terminal actual.

> 💡 `Nota de control`: En este modo, si cierras la terminal o presionas `Ctrl + C`, enviarás una señal de interrupción
> al proceso principal de Spring Boot, lo que causará que el contenedor se detenga.

### 📦 Desatachar deteniendo el contenedor

Si queremos volver a tener el control de la línea de comando debemos presionar `Ctrl + C`, pero **esto detendrá el
contenedor**, cambiando su status `Up` por `Exited`.

````bash
$ docker container ls -a
CONTAINER ID   IMAGE          COMMAND                  CREATED          STATUS                       PORTS     NAMES
d422a460f491   user-service   "java org.springfram…"   14 minutes ago   Exited (130) 5 minutes ago             happy_archimedes
````

### 📦 Desatachar sin detener el contenedor

Si tu terminal quedó atachada al contenedor después de ejecutar `docker container run`, puedes recuperar el control de
la terminal sin detener el contenedor usando la siguiente combinación de teclas.

````bash
Ctrl + P
Ctrl + Q
````

**💡 Nota**
> A mí no me funcionó la combinación de teclas, pero en la mayoría de las fuentes consultadas mencionan esa combinación.
> Así que para evitar problemas lo mejor será que cuando cree un contenedor agregarle el flag `-d` para que siempre se
> cree `desatachado`.

### 📦 Crea contenedor con el flag -d (detach)

Si ejecutamos un contenedor usando el flag `-d` estaremos ejecutándolo en modo `dettached`, es decir, el contenedor
se ejecutará en segundo plano, sin que la terminal quede `atachada` a él. Esto nos permite seguir usando la terminal
para otros comandos, mientras el contenedor sigue corriendo en segundo plano.

````bash
$ docker container run -d -p 8001:8001 user-service
7aa9f5d8fda50ca1ba06474c6034334cd761f057bc4065876b343233cd629846
````

### 📦 Asignación de nombre al contenedor `(--name)`

Por defecto, si no especificamos un nombre al crear un contenedor, Docker le asignará uno aleatorio compuesto por un
adjetivo y el apellido de un científico o ingeniero famoso (por ejemplo, `loving_curie` o `suspicious_rhodes`).

Para mantener un control profesional y facilitar la administración, utilizamos la bandera `--name`, lo que nos permite
identificar al contenedor de forma unívoca y descriptiva.

En este ejemplo, bautizaremos a nuestro contenedor como `c-user-service`:

````bash
$ docker container run -d -p 9001:8001 --rm --name c-user-service user-service
0e253320abccf08a874d3e1ae0ac97dacaac38c60329f99158eb1b5b936aa3d9
````

Al listar los contenedores, observamos que en la columna `NAMES` aparece el identificador que hemos elegido:

````bash
$ docker container ls -a
CONTAINER ID   IMAGE          COMMAND                  CREATED          STATUS          PORTS                                         NAMES
0e253320abcc   user-service   "java org.springfram…"   14 seconds ago   Up 13 seconds   0.0.0.0:9001->8001/tcp, [::]:9001->8001/tcp   c-user-service
````

### 📦 Lista todos los contenedores

El siguiente comando se utiliza para listar todos los contenedores en tu sistema, incluyendo tanto los que
están en ejecución como los que han sido detenidos. La opción `-a` significa `all` (todos), y nos muestra el estado
completo de los contenedores, no solo los que están activos.

````bash
$ docker container ls -a
CONTAINER ID   IMAGE          COMMAND                  CREATED              STATUS              PORTS                                         NAMES
7aa9f5d8fda5   user-service   "java org.springfram…"   About a minute ago   Up About a minute   0.0.0.0:8001->8001/tcp, [::]:8001->8001/tcp   loving_stonebraker
````

- `docker container ls`: Muestra solo los contenedores que están en ejecución en ese momento.
- `docker container ls -a`: Muestra todos los contenedores, independientemente de si están en ejecución, detenidos
  o en estado creado pero nunca ejecutados.

### 📦 Detención de contenedor en ejecución

Para detener uno o más contenedores que se encuentran actualmente en marcha, utilizamos el comando `stop`. Docker
enviará una señal de terminación controlada `(SIGTERM)` al proceso principal (en este caso, a nuestra aplicación Spring
Boot), permitiéndole cerrar conexiones y finalizar tareas pendientes antes de apagarse.

Para identificar el contenedor a detener, podemos utilizar tanto el `CONTAINER_ID` como el nombre asignado en
la columna `NAMES`.

````bash
$ docker container stop 7aa9f5d8fda5
7aa9f5d8fda5
````

Una vez ejecutado el comando, si listamos todos los contenedores (incluidos los inactivos), observaremos que el estado
ha cambiado a `Exited`.

````bash
$ docker container ls -a
CONTAINER ID   IMAGE          COMMAND                  CREATED         STATUS                        PORTS     NAMES
7aa9f5d8fda5   user-service   "java org.springfram…"   3 minutes ago   Exited (143) 39 seconds ago             loving_stonebraker
````

### 📦 Inicia contenedor detenido

Con el siguiente comando iniciamos uno o más contenedores detenidos.

````bash
$ docker container start 7aa9f5d8fda5
7aa9f5d8fda5
````

### 📦 Eliminación automática al detenerse `(--rm)`

Por defecto, cuando un contenedor se detiene, su instancia permanece en el sistema en estado `Exited`.

### 📦 Modo interactivo al crear un nuevo contenedor `(run -it)`

A diferencia de `exec` (que actúa sobre algo ya vivo), aquí creamos un contenedor desde cero e ingresamos directamente
a él para inspeccionarlo. Al combinarlo con la bandera `--rm`, garantizamos que no quede rastro del contenedor
una vez que terminemos la inspección.

#### 🛠️ Ejecución con sobreescritura de `ENTRYPOINT`

Si nuestra imagen tiene un `ENTRYPOINT` definido (como el que lanza nuestra aplicación `Spring Boot`),
no podemos simplemente añadir `/bin/sh` al final del comando. Debemos usar la bandera específica `--entrypoint`
para forzar el cambio:

````bash
$ docker container run -p 9001:8001 --rm -it --entrypoint /bin/sh user-service
/app # ls -l
total 12
drwxr-xr-x 1 root root 4096 Mar 25 21:17 BOOT-INF
drwxr-xr-x 4 root root 4096 Mar 25 21:17 META-INF
drwxr-xr-x 3 root root 4096 Mar 25 20:59 org
/app # 
````

- `--entrypoint /bin/sh`: sobrescribe completamente el `ENTRYPOINT` definido en tu `Dockerfile`
  (`["java", "org.springframework.boot.loader.launch.JarLauncher"]`). En este escenario,
  `Spring Boot no arrancará`, ya que el proceso principal ahora es el `shell de comandos`. Esto significa que:
    - El contenedor arranca directamente con `/bin/sh`.
    - El comando original (`java ... JarLauncher`) no se ejecuta en absoluto.
    - Es como si no existiera, porque `Docker` lo reemplaza en tiempo de ejecución.
    - 👉 En otras palabras:
        - `ENTRYPOINT` en `Dockerfile` = valor por defecto.
        - `--entrypoint` en `docker run` = reemplazo total del valor por defecto.
    - Por eso al entrar ves el prompt `/app #`, ya que estás dentro de un shell interactivo en el contenedor,
      y no en tu aplicación Java.


- `-it`: Nos otorga la terminal interactiva necesaria para escribir comandos dentro.
- `--rm`: Marca el contenedor como "desechable". En cuanto salgamos, Docker lo borrará automáticamente.

💡 Nota: A diferencia de `ENTRYPOINT`, el `CMD` sí puede ser reemplazado directamente. Por ejemplo, si la imagen
utilizara `CMD`, bastaría con ejecutar:

````bash
$ docker container run -p 9001:8001 --rm -it user-service /bin/sh
````

#### ⚖️ Diferencia entre `ENTRYPOINT` y `CMD`

Es vital entender cómo se comportan estas dos instrucciones al momento de ejecutar `docker run`:

| Instrucción en Dockerfile | ¿Cómo se sobrescribe en el `run`?              | Ejemplo de comando                       |
|---------------------------|------------------------------------------------|------------------------------------------|
| **`CMD`**                 | Basta con añadir el comando al final.          | `docker run imagen /bin/sh`              |
| **`ENTRYPOINT`**          | Requiere la bandera específica `--entrypoint`. | `docker run --entrypoint /bin/sh imagen` |

#### 🚪 Salida y limpieza automática

Mientras estemos dentro, el contenedor aparecerá en estado `Up` en cualquier otra terminal:

````bash
$ docker container ls -a
CONTAINER ID   IMAGE          COMMAND     CREATED         STATUS         PORTS                                         NAMES
0df5269effb7   user-service   "/bin/sh"   6 minutes ago   Up 6 minutes   0.0.0.0:9001->8001/tcp, [::]:9001->8001/tcp   suspicious_rhodes
````

Al escribir `exit`, el proceso `/bin/sh` termina. Como es el proceso principal `(PID 1)`, el contenedor se detiene y,
gracias a `--rm`, desaparece inmediatamente:

````bash
$ docker container run -p 9001:8001 --rm -it --entrypoint /bin/sh user-service
/app # ls -l
total 12
drwxr-xr-x 1 root root 4096 Mar 25 21:17 BOOT-INF
drwxr-xr-x 4 root root 4096 Mar 25 21:17 META-INF
drwxr-xr-x 3 root root 4096 Mar 25 20:59 org
/app # exit
````

Comprobamos que el contenedor se eliminó automáticamente:

````bash
$ docker container ls -a
CONTAINER ID   IMAGE     COMMAND   CREATED   STATUS    PORTS     NAMES
````

#### ⚠️ Notas Críticas de Funcionamiento

Al sobrescribir el `ENTRYPOINT` con `/bin/sh`, el proceso principal `(PID 1)` pasa a ser el `shell de comandos` y no la
aplicación `Spring Boot`. Como resultado:

- `La aplicación Java no se ejecuta`: El contexto de Spring nunca llega a cargar.
- `Puertos Inactivos`: Aunque Docker reserve y mapee los puertos `9001:8001`, no habrá ningún servicio escuchando
  dentro del contenedor. Cualquier petición a `localhost:9001` fallará.

## 📂 Copia de archivos y directorios

### ⬆️ Desde Máquina Local hacia el Contenedor

El comando `cp (copy)` es una herramienta bidireccional que permite mover archivos o carpetas entre el sistema de
archivos de tu `computadora (Host)` y un `contenedor activo`.

#### 1. Preparación del entorno

Primero, nos aseguramos de tener un contenedor en ejecución:

````bash
$ docker container run -d -p 8001:8001 --rm user-service
fdf46d24153c5ceaf11c5c71fef545b0cf8cd9067c529f5c3b5620f56570759d
````

#### 2. Localización del archivo origen

Desde nuestra terminal local (en este caso, en la unidad `M:\`), verificamos la existencia del archivo que deseamos
transferir(`mi-archivo.txt`):

````bash
M:\
$ ls -l
total 1
drwxr-xr-x 1 magadiflo 197121  0 Apr  4  2025 '$RECYCLE.BIN'/
-rw-r--r-- 1 magadiflo 197121 28 Mar 26 18:13  mi-archivo.txt
drwxr-xr-x 1 magadiflo 197121  0 Apr  4  2025 'System Volume Information'/
````

#### 3. Ejecución de la copia

Utilizamos la sintaxis `docker container cp [ORIGEN] [DESTINO]`. Para el destino, usamos el formato
`ID_CONTENEDOR:/ruta/interna`.

````bash
M:\
$ docker container cp .\mi-archivo.txt fdf46d24153c:/app
Successfully copied 2.05kB to fdf46d24153c:/app
````

Análisis del comando:

- `cp`: Orden de copiado de archivos/carpetas.
- `.\mi-archivo.txt`: Ruta del archivo en nuestra máquina física.
- `fdf46d24153c:/app`: Identificador del contenedor seguido de la ruta absoluta donde se depositará el archivo.

#### 4. Verificación interna

Finalmente, ingresamos al contenedor para confirmar que el archivo ha llegado a su destino y verificar su integridad:

````bash
$ docker container exec -it fdf46d24153c /bin/sh
/app # ls -l
total 16
drwxr-xr-x 1 root root 4096 Mar 25 21:17 BOOT-INF
drwxr-xr-x 4 root root 4096 Mar 25 21:17 META-INF
-rwxr-xr-x 1 root root   28 Mar 26 23:13 mi-archivo.txt
drwxr-xr-x 3 root root 4096 Mar 25 20:59 org

/app # cat mi-archivo.txt
Este es mi archivo de prueba/app #
````

#### 💡 Datos importantes sobre cp

- `Estado del contenedor`: A diferencia de otros comandos, `cp` funciona tanto con contenedores en ejecución como con
  contenedores detenidos.
- `Permisos`: Docker intentará preservar los permisos de usuario originales, aunque dentro del contenedor el archivo
  suele ser propiedad del usuario que ejecuta el proceso principal (normalmente root).
- `Copia de Directorios`: Si deseas copiar una carpeta completa, la sintaxis es la misma:
  `docker container cp ./mi-carpeta id_contenedor:/ruta`.

> `Uso recomendado`: Este comando es excelente para inyectar configuraciones rápidas, archivos de licencias o
> logs de prueba, pero recuerda que si el contenedor se elimina (especialmente usando `--rm`), estos archivos
> copiados desaparecerán con él. Para persistencia real, se recomienda el uso de `Volumes`.

## 📝 Copia de Logs de Spring Boot al Host

Extraer archivos de log físicos es una práctica común para realizar auditorías o análisis detallados fuera del entorno
de ejecución.

### [⚙️ Configuración de registro predeterminada en Spring Boot](https://reflectoring.io/springboot-logging/)

La configuración de `logging` predeterminada en Spring Boot es una implementación de `Logback` en el nivel de `info`
para `registrar la salida en la consola`.

### [🗂️ Logging en un archivo](https://reflectoring.io/springboot-logging/)

Podemos redirigir los logs a un archivo configurando una de estas dos propiedades en `application.properties`:

- `logging.file.name` → define el nombre exacto del archivo.
- `logging.file.path` → define la carpeta donde se generará un archivo llamado `spring.log`.

> ⚠️ `Importante`: si configuramos ambas, solo tendrá efecto `logging.file.name`.

### 📄 Ejemplos de configuración

#### - `Opción 1`: Personalizar el nombre del archivo de log como `application.log`.

````properties
# Salida a un archivo llamado application.log
logging.file.name=application.log
````

#### - `Opción 2`: Generar el archivo `spring.log` en una ruta específica.

````properties
# Salida a un archivo llamado spring.log en la ruta /Users
logging.file.path=/Users
````

### 1. Configuración de Logging en Spring Boot

Por defecto, Spring Boot escribe logs en la consola. Para dirigirlos a un archivo, configuramos la propiedad
`logging.file.name` en nuestro `application.yml` (Es la opción que elegiré).

````yml
spring:
  application:
    name: user-service

logging:
  level:
    dev.magadiflo.user.app: debug
    org.hibernate.SQL: debug
  file:
    # Ruta absoluta dentro del contenedor
    name: /app/logs/${spring.application.name}.log
````

> Al usar `${spring.application.name}`, el archivo resultante tomará el nombre de nuestra aplicación definido en
> `spring.application.name` más la concatenación del `.log`.

### 2. Preparación en el Dockerfile

Dado que `Spring Boot` intentará escribir en `/app/logs`, debemos asegurarnos de que ese directorio exista dentro de la
imagen. Lo creamos en la etapa final `(runner)`:

````dockerfile
FROM eclipse-temurin:25-jdk-alpine AS dependencies
WORKDIR /app
COPY ./mvnw ./
COPY ./.mvn ./.mvn
COPY ./pom.xml ./
RUN sed -i -e 's/\r$//' ./mvnw && ./mvnw dependency:go-offline
COPY ./src ./src
RUN ./mvnw clean package -DskipTests
````

### ✅ Puntos clave a recordar

- `Persistencia de la Copia`: Una vez ejecutado el comando `docker container cp`, los archivos en tu máquina local
  `(M:\files)` están seguros y persistentes. Si el contenedor se detiene o se elimina, los archivos que ya copiaste no
  se verán afectados.
- `Archivos No Copiados`: Solo conservas lo que extrajiste manualmente. Cualquier log nuevo que la aplicación genere
  dentro del contenedor después de tu última copia, se eliminará permanentemente si el contenedor se destruye
  (especialmente con `--rm`).
- `Hacia Producción (Volumes)`: En entornos reales, para no tener que estar ejecutando `cp` manualmente a cada rato, se
  suelen usar `Volumes`. Esto crea un "puente" en tiempo real entre la carpeta del contenedor y una carpeta de tu
  servidor, de modo que cada línea de log que escribe Spring Boot aparece instantáneamente en tu máquina.
