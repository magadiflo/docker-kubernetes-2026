# 🐳 Sección 06: Docker - Introducción

`Docker` no es solo una herramienta de virtualización; es el estándar de la industria para garantizar que el código que
escribiste en tu máquina funcione exactamente igual en el servidor de producción. A continuación, desglosamos su
arquitectura y componentes esenciales.

---

## 🧐 [¿Qué es docker?](https://docs.docker.com/get-started/docker-overview/)

`Docker` es una plataforma de software que automatiza el despliegue de aplicaciones dentro de contenedores.
Su propuesta de valor principal es la `eliminación del "En mi máquina funciona"`, permitiendo separar la aplicación de
la infraestructura subyacente.

### 🌟 La Propuesta de Valor

Al utilizar Docker para tus microservicios de Java/Spring Boot, obtienes:

- `Portabilidad Extrema`: El mismo artefacto se ejecuta en Windows, Linux o la Nube.
- `Aislamiento`: Cada microservicio (`user-service`, `course-service`) tiene sus propias dependencias sin entrar en
  conflicto con el sistema host.
- `Eficiencia`: A diferencia de las Máquinas Virtuales (VM), Docker comparte el kernel del sistema operativo host,
  lo que lo hace increíblemente ligero.

## 🏗️ Arquitectura de Docker: El Modelo Cliente-Servidor

Docker opera bajo una arquitectura desacoplada donde el cliente y el servidor se comunican a través de una API REST.

### 😈 1. El Demonio de Docker (`dockerd`)

Es el "cerebro" del sistema. Se ejecuta en segundo plano y es el responsable de:

- Gestionar el ciclo de vida de los `Contenedores`.
- Almacenar y construir `Imágenes`.
- Administrar `Redes` (para que los microservicios hablen entre sí).
- Gestionar `Volúmenes` (para que los datos no se pierdan).

### 💻 2. El Cliente Docker (`docker`)

Es la interfaz de línea de comandos (CLI) que utilizamos nosotros. Cuando escribimos un comando como
`docker container run...`, el cliente lo traduce a una petición API y se la envía al `dockerd`.

> `Dato Pro`: El cliente puede conectarse a un `demonio local` o a uno `remoto` situado en otra red.

### 🏪 3. Docker Registry (`El Almacén`)

Lugar donde se `guardan las imágenes de Docker`.

- `Docker Hub` es un registro público que cualquiera puede usar. Además, `Docker` busca imágenes en
  `Docker Hub de forma predeterminada`.
- `Privados`: Empresas suelen usar AWS ECR, Azure ACR o su propio GitLab Registry para proteger su código.

Cuando usas los comandos `docker pull` o `docker run`, `Docker` extrae las imágenes necesarias de tu registro
configurado. Cuando usas el comando `docker push`, `Docker` envía tu imagen a tu registro configurado.

## 🧩 Componentes Clave: Objetos de Docker

Para entender Docker, debemos diferenciar claramente entre sus dos componentes principales:

### 💿 Imágenes: La Plantilla Estática

Una `imagen` es una plantilla de solo lectura con instrucciones para crear un `contenedor Docker`. A menudo, una
`imagen` se basa en otra `imagen`, con alguna personalización adicional. Por ejemplo, puede crear una imagen que
se base en la imagen de Ubuntu, pero que instale el servidor web Apache y su aplicación, así como los detalles
de configuración necesarios para que su aplicación se ejecute.

Puede crear sus propias imágenes o puede utilizar únicamente las creadas por otros y publicadas en un registro.
`Para crear su propia imagen`, debe crear un `Dockerfile` con una sintaxis simple para definir los pasos necesarios
para crear la imagen y ejecutarla.

- `Cada instrucción en un Dockerfile crea una capa en la imagen`. Cuando cambia el `Dockerfile` y reconstruye la imagen,
  solo se reconstruyen las capas que han cambiado. Esto es parte de lo que hace que las imágenes sean tan livianas,
  pequeñas y rápidas, en comparación con otras tecnologías de virtualización.
- `Inmutabilidad`: Una vez creada, la imagen no cambia.

### 📦 Contenedores: La Instancia Viva

`Un contenedor es una instancia ejecutable de una imagen`. Puede crear, iniciar, detener, mover o eliminar un
contenedor mediante la API o la CLI de Docker. Puede conectar un contenedor a una o más redes, adjuntarle
almacenamiento o incluso crear una nueva imagen en función de su estado actual.

- Es la unidad ejecutable.
- Es `efímero`: Si el contenedor se borra, cualquier dato guardado dentro de él (que no esté en un volumen) desaparece.
- `Aislamiento`: de manera predeterminada,
  `un contenedor está relativamente bien aislado de otros contenedores y de su máquina host`.
  Puede controlar el grado de aislamiento de la red, el almacenamiento u otros subsistemas subyacentes de un
  contenedor respecto de otros contenedores o de la máquina host.

## ☕ La Analogía del Desarrollador Java

| Concepto en Java                      | Concepto en Docker                   | Característica                                                                                                     |
|---------------------------------------|--------------------------------------|--------------------------------------------------------------------------------------------------------------------|
| **Clase (`.class`)**                  | **Imagen**                           | Es el molde estático. Define el comportamiento y las propiedades, pero no "hace" nada por sí sola.                 |
| **Instancia (`new Object()`)**        | **Contenedor**                       | Es la ejecución en memoria. Puedes crear cientos de objetos a partir de una sola clase; cada uno es independiente. |
| **Atributos (`private String name`)** | **Variables de Entorno / Volúmenes** | Definen el estado específico de esa instancia particular (ej. la IP de la base de datos).                          |
| **JAR / Maven Repository**            | **Docker Hub / Registry**            | Donde guardas y distribuyes tus "clases" (imágenes) para que otros las reutilicen.                                 |

#### 💡 ¿Qué es el archivo `.jar.original`?

Al empaquetar la aplicación, `Maven` genera primero un `JAR estándar` de Java (el `.original`).
Inmediatamente después, el `plugin de Spring Boot` lo re-empaqueta para convertirlo en un `"Fat JAR"` ejecutable,
integrando un cargador de clases propio y todas las dependencias necesarias en la carpeta interna `BOOT-INF/lib`.

#### 🎯 Nuestra elección: El `Fat JAR ejecutable` `(.jar)`

Este archivo es el resultado final del proceso de `re-empaquetado`. A diferencia del estándar, este es un artefacto
autónomo que contiene no solo nuestro código compilado, sino también el servidor embebido (Tomcat) y todas las
librerías externas que el microservicio necesita para funcionar.

Lo utilizamos para `Dockerizar` porque garantiza la portabilidad absoluta: el contenedor no necesita que instalemos
librerías ni servidores externos en el sistema operativo; todo lo necesario para iniciar el servicio ya vive dentro de
este archivo.

En ese sentido, el archivo que nos interesa para Docker es el `re-empaquetado` y `ejecutable`, que se identifica por
ser el más pesado y terminar simplemente en `.jar`:

> `user-service-0.0.1-SNAPSHOT.jar`

#### 💡 Reflexión: ¿Por qué este paso es vital para Docker?

Este ejercicio nos permite identificar las dependencias externas y los puntos críticos que Docker deberá resolver por
nosotros:

1. `Independencia del IDE`: Confirmamos que no necesitamos `IntelliJ` o `Eclipse` para correr la app. Solo necesitamos
   el `JAR` y el `JRE` (Java Runtime Environment).
2. `El "Problema de la Máquina"`: Actualmente, para que este comando funcione en otra computadora, tendríamos que
   instalar manualmente:
    - La misma versión de Java (v25 en este caso).
    - Configurar las variables de entorno.
    - Asegurar que la Base de Datos sea accesible desde esa máquina.
3. `Hacia la Contenedorización`: Docker tomará este mismo proceso (copiar el JAR y ejecutar `java -jar`) pero lo hará
   dentro de un entorno aislado donde `él mismo instalará el Java correcto`, garantizando que funcione en cualquier
   lugar sin que nosotros instalemos nada manualmente en el host.

### 🚀 [Paso 3: Creación del Dockerfile para user-service](https://spring.io/guides/gs/spring-boot-docker)

Docker utiliza un formato de archivo sencillo llamado `Dockerfile` para especificar las `capas` de una imagen.
Creamos un archivo llamado exactamente `Dockerfile` (sin extensión) en la raíz de `user-service`:

````dockerfile
# 1. Imagen base con el JDK de Java 25 (Versión ligera Alpine)
FROM eclipse-temurin:25-jdk-alpine

# 2. Directorio de trabajo interno
WORKDIR /app

# 3. Copia del artefacto (Fat JAR) al contenedor
COPY ./target/*.jar ./app.jar

# 4. Documentación del puerto de escucha
EXPOSE 8001

# 5. Comando de ejecución principal
ENTRYPOINT ["java", "-jar", "app.jar"]
````

#### 🔍 Desglose Técnico de las Instrucciones

- 🐋 `FROM eclipse-temurin:25-jdk-alpine` — es nuestro punto de partida.
    - **Imagen:** `Eclipse Temurin`, la distribución de `OpenJDK` más recomendada por la comunidad por su estabilidad
      y soporte a largo plazo.
    - **Tag:** `25-jdk-alpine`, basado en `Alpine Linux`, una distribución ultra ligera (menos de 5 MB). Usar `Alpine`
      reduce drásticamente el tamaño de la imagen final y minimiza la superficie de ataque de seguridad.
    - Imagen disponible en: https://hub.docker.com/_/eclipse-temurin/tags?name=25-jdk-alpine


- 📂 `WORKDIR /app` — define el directorio de trabajo de nuestra aplicación dentro del contenedor.
    - No es obligatorio, pero su uso evita que los archivos de la aplicación se mezclen con los directorios del sistema
      operativo. Todas las instrucciones siguientes (`COPY`, `ENTRYPOINT`) se ejecutarán tomando `/app` como base.


- 📋 `COPY ./target/*.jar ./app.jar` — copia el artefacto generado en nuestra máquina local hacia la imagen.
    - La ruta de origen `./target/*.jar` apunta al directorio `target` de nuestra máquina local, donde Maven genera
      el JAR al compilar el proyecto.
    - Se usa el comodín `*.jar` en lugar del nombre completo `user-service-0.0.1-SNAPSHOT.jar` para evitar
      actualizarlo cada vez que cambie la versión del proyecto. Siempre habrá un único archivo `.jar` en ese
      directorio, así que el comodín es seguro.
    - En el destino, el archivo se renombra a `app.jar`, un nombre genérico e independiente de la versión.
    - El punto inicial `./app.jar` es relativo al `WORKDIR`, por lo que la ruta absoluta final dentro del
      contenedor será `/app/app.jar`.


- 📄 `EXPOSE 8001` — instrucción meramente informativa `(opcional)`.
    - No publica ni abre el puerto en el host. Su propósito es documentar qué puerto usa internamente el contenedor,
      sirviendo como contrato entre quien construye la imagen y quien ejecuta el contenedor.
    - El puerto indicado es el `puerto interno del contenedor`, es decir, el puerto donde la aplicación escucha
      dentro de él. El mapeo hacia el host se define al ejecutar el contenedor con `-p`.


- ▶️ `ENTRYPOINT ["java", "-jar", "app.jar"]` — define el comando que se ejecutará al iniciar el contenedor.
    - Se ejecuta tomando el `WORKDIR` como directorio base, es decir, desde `/app`.
    - **Es una instrucción para contenedores, no para imágenes.** No tiene efecto durante la construcción de la imagen
      sino al momento de ejecutar el contenedor.
    - Los argumentos pasados con `docker run` se añaden al comando definido en `ENTRYPOINT`, sin sobrescribirlo.

#### [¿Qué son `CMD` y `ENTRYPOINT`?](https://devtron.ai/blog/cmd-and-entrypoint-differences/)

`CMD` y `ENTRYPOINT` son instrucciones de `Dockerfile` que definen el comportamiento del contenedor al iniciarse.
Ambas especifican qué comando debe ejecutarse, pero cumplen funciones diferentes.

> **📌 Importante:** Cuando hablamos de "argumentos" en este contexto, nos referimos a los que se colocan
> **después del nombre de la imagen** en el comando `docker run`, no a las flags propias de Docker como
> `--name`, `-p` o `--env`. La anatomía es la siguiente:
> ```bash
> docker container run [flags de Docker] my-app [argumentos para el contenedor]
> #                     ↑                        ↑
> #               --name, -p, --env         estos son los que afectan
> #               Docker los consume        al CMD o se añaden al ENTRYPOINT
> #               antes de arrancar
> ```

- **`CMD`**
    - Define el comando predeterminado del contenedor. Si no se pasan argumentos tras el nombre de la imagen,
      se ejecuta lo definido en `CMD`.
    - Si se pasan argumentos en `docker run`, estos **sobrescriben completamente** el `CMD`. Por ejemplo, si
      tenemos `CMD ["java", "-jar", "app.jar"]` y ejecutamos:
        ```bash
        docker container run my-app --spring.profiles.active=dev
        ```
    - El comando resultante sería solo `--spring.profiles.active=dev`, perdiendo el `java -jar app.jar`
      por completo — lo que provocaría un fallo en el contenedor.


- **`ENTRYPOINT`**
    - Define el ejecutable principal del contenedor. Los argumentos pasados tras el nombre de la imagen
      en `docker run` se **añaden** al comando del `ENTRYPOINT`, sin sobrescribirlo. Por ejemplo, con
      `ENTRYPOINT ["java", "-jar", "app.jar"]`:
        ```bash
        docker container run my-app --spring.profiles.active=dev
        ```
    - El comando resultante sería `java -jar app.jar --spring.profiles.active=dev` — el `ENTRYPOINT`
      se respeta y el argumento simplemente se anexa al final.
    - Solo puede sobrescribirse explícitamente usando la flag `--entrypoint` en `docker run`.

### 🛡️ ¿Por qué elegimos `ENTRYPOINT`?

Para nuestro `user-service`, no queremos que el contenedor haga otra cosa que no sea ejecutar Java.

1. `Protección`: Evita que alguien accidentalmente ejecute el contenedor y lo deje inactivo
   (por ejemplo, si sobrescriben el comando por error).
2. `Inmutabilidad`: Define el propósito fijo del contenedor: `Ejecutar nuestro Microservicio`.

`Patrón Pro`:
> Si usáramos ambos, `ENTRYPOINT` sería el comando fijo y `CMD` contendría argumentos predeterminados
> pero flexibles:
> ```dockerfile
> ENTRYPOINT ["java", "-jar", "app.jar"]
> CMD ["--spring.profiles.active=prod"]
> ```
> Así, el JAR **siempre** se ejecuta (gracias a `ENTRYPOINT`), el perfil por defecto es `prod`
> (gracias a `CMD`), pero puede sobrescribirse fácilmente al ejecutar el contenedor:
> ```bash
> docker container run my-app --spring.profiles.active=dev
> ```
> Resultando en: `java -jar app.jar --spring.profiles.active=dev`

Para profundizar con ejemplos prácticos, visita:
[Understanding CMD and ENTRYPOINT Differences in Docker](https://devtron.ai/blog/cmd-and-entrypoint-differences/)

### 🔌 Conectando Mundos: El Desafío del `localhost`

Antes de construir la imagen, debemos resolver un problema de comunicación crítico. En Docker, cada contenedor
tiene su propia interfaz de red. Por lo tanto:

- **En tu PC:** `localhost` es tu máquina física.
- **En el Contenedor:** `localhost` es el propio contenedor, `no tu PC`.

Si dejamos la configuración actual, nuestro `user-service` buscará la base de datos dentro del contenedor
(donde no hay ningún MySQL instalado) y la aplicación fallará al iniciar.

### 🌐 La Solución: `host.docker.internal`

Para permitir que el microservicio (dentro del contenedor) "salte" hacia afuera y encuentre el MySQL de tu
máquina local (Host), Docker nos proporciona un nombre de dominio especial:

> `host.docker.internal`: Es un DNS interno que resuelve a la dirección IP de tu máquina host desde la perspectiva del
> contenedor.

### 📝 Actualización del `application.yml`

Modificamos la URL de conexión en nuestro microservicio `user-service`:

````yml
spring:
  application:
    name: user-service
  datasource:
    # Cambiamos localhost por el DNS especial de Docker
    url: jdbc:mysql://host.docker.internal:3306/db_user_service
    username: admin
    password: magadiflo
````

## 🔄 Comunicación Híbrida: Contenedor ↔️ Máquina Local

En esta etapa del curso, nos enfrentamos a un escenario común en la transición a Docker: tenemos un microservicio ya
dockerizado `(user-service)` que necesita hablar con otro que aún corre de forma nativa en nuestra PC
`(course-service)`.

### 🚀 1. El desafío de las URLs de comunicación

La clave para que esto funcione es entender quién inicia la conexión:

### 🐳 En el user-service (Contenedor)

Como el `user-service` está "encerrado" en Docker, no puede usar `localhost` para llegar al `course-service`,
ya que eso lo redirigiría a sí mismo.

- Solución: Usamos nuevamente el DNS especial de Docker `host.docker.internal`.
- Configuración en `application.yml` de `user-service`:

````yml
custom:
  course-service:
    # host.docker.internal resuelve a la IP de nuestra PC desde el contenedor
    base-url: http://host.docker.internal:8002/api/v1/courses
````

### 🟢 En el course-service (Máquina Local)

Desde la perspectiva del `course-service` (que corre de forma nativa en el Host), el contenedor del `user-service`
se comporta como un servicio local estándar. Esto es posible gracias al mapeo de puertos que realizamos al lanzar
el contenedor (`-p 8001:8001`).

Al haber expuesto el puerto, el `course-service` simplemente utiliza `localhost` y el puerto externo definido `(8001)`
para comunicarse con el contenedor, sin necesidad de configuraciones especiales de red.

Esta configuración se encuentra en el `application.yml` del `course-service`:

````yml
custom:
  user-service:
    base-url: http://localhost:8001/api/v1/users
````
