## 🛠️ Creación del archivo `.dockerignore`

Antes de mejorar el `Dockerfile`, debemos controlar qué información le enviamos a `Docker`. Por defecto,
cuando ejecutamos `docker image build .`, Docker toma todos los archivos de la carpeta actual y los envía al motor
`(Daemon)` de Docker. Esto se conoce como el `Contexto de Construcción`.

### 🧐 ¿Por qué es necesario el `.dockerignore`?

Sin este archivo, estaríamos enviando carpetas pesadas e innecesarias al Daemon, como:

- `target/`: Que contiene compilaciones antiguas.
- `.git/`: Todo el historial de versiones.
- `.idea/` o `.vscode/`: Configuraciones personales de tu editor.
- `Archivos temporales`: Logs o reportes de tests.

Esto no solo hace que el proceso de build sea más lento, sino que también puede aumentar el tamaño de la imagen final y
exponer archivos sensibles.

El archivo `.dockerignore` es un archivo de texto que especifica qué archivos y directorios deben ser ignorados al
construir una imagen de `Docker`. Funciona de manera similar a un archivo `.gitignore` en `Git`, ayudando a excluir
ciertos archivos innecesarios del contexto de construcción (es decir, el conjunto de archivos que `Docker` envía al
daemon de `Docker` para crear una imagen).

### 📝 Configuración de "Lista Blanca" (Whitelist)

En lugar de listar cada carpeta que queremos ignorar, utilizaremos una estrategia de `exclusión total` y
`rescate selectivo`. Crearemos un archivo `.dockerignore` en la raíz de cada microservicio
(`user-service` y `course-service`):

````dockerignore
# 1. Bloqueo total: Ignora todos los archivos y directorios del contexto
*
# 2. Rescate selectivo: Solo estos elementos son necesarios para construir la app
!src/
!pom.xml
!.mvn/
!mvnw
````

#### 🔍 Análisis de la Estrategia:

- El asterisco `(*)`: Le dice a Docker: "No te lleves nada".
- El signo de exclamación `(!)`: Es una excepción. Le dice a Docker: "A pesar de la regla anterior, SÍ necesito estos
  archivos".
    - `src/`: Nuestro código fuente.
    - `pom.xml`: Nuestra receta de Maven.
    - `.mvn/` y `mvnw`: Las herramientas para compilar sin depender de un Maven instalado en el host.

## 🏗️ Optimizando Dockerfile 01: Construcción Automática (In-Container Build)

Hasta ahora, nuestro flujo dependía de un paso previo en la máquina host (generar manualmente el `jar`). Con esta
optimización, el `Dockerfile` se vuelve autosuficiente: `él recibe el código y él entrega el ejecutable`.

### 📝 El nuevo `Dockerfile` Autogestionado

````dockerfile
# 1. Imagen base con el JDK de Java 25
FROM eclipse-temurin:25-jdk-alpine

# 2. Directorio de trabajo
WORKDIR /app

# 3. Copiamos el contexto filtrado por .dockerignore (src, pom.xml, mvnw)
COPY ./ ./

# 4. Saneamiento del script y compilación
RUN sed -i -e 's/\r$//' ./mvnw && ./mvnw clean package -DskipTests

# 5. Puerto de escucha de Spring Boot
EXPOSE 8001

# 6. Ejecución del JAR generado internamente
ENTRYPOINT ["java", "-jar", "./target/user-service-0.0.1-SNAPSHOT.jar"]
````

### 🔍 Análisis de las Nuevas Instrucciones

#### 📥 `COPY ./ ./`

Gracias al archivo `.dockerignore` que creamos en el paso anterior, esta instrucción ya no es "peligrosa". Docker solo
copiará el código fuente y los archivos de configuración de Maven, ignorando carpetas pesadas como `target/` local o
`.git/`.

Entonces, la instrucción copia todos los archivos y directorios no ignorados del directorio actual (`./`)
al directorio de trabajo dentro del contenedor (segundo `./` es equivalente a `/app`).

#### 🧹 El comando `sed` (Saneamiento de Windows a Linux)

Esta es una técnica de robustez.

- **El Problema:** Si trabajas en Windows, el archivo `./mvnw` tiene finales de línea `CRLF (\r\n)`.
  Al copiarlo a una imagen basada en `Linux (Alpine)`, el sistema no reconoce el comando y lanza errores extraños.
- **La Solución:** `sed -i -e 's/\r$//' ./mvnw` elimina quirúrgicamente el carácter invisible `\r`, asegurando que el
  script sea ejecutable en el entorno Linux del contenedor.
    - `Descripción`: El comando `sed` se utiliza para eliminar los caracteres de retorno de carro
      (`\r`, que son típicos en sistemas Windows) del archivo `mvnw`.
    - `Propósito`: Evitar problemas de compatibilidad en sistemas basados en `Unix/Linux` (como `Alpine`) cuando se
      ejecuta el script `mvnw`.
    - `-i`: Es una opción que le indica a sed que realice cambios en el archivo en su lugar, es decir, modificará el
      archivo `mvnw` directamente.
    - `-e`: Indica que se proporcionará una expresión de script a `sed`.
    - `'s/\r$//'`: Es la expresión de script que busca y reemplaza el retorno de carro (`\r`) al final de cada línea
      del archivo por una cadena vacía, es decir, lo elimina.
    - `./mvnw`: Es el archivo en el que se realizará la modificación. En este caso, se asume que el archivo `mvnw`
      está en el directorio actual (`./` que es equivalente al WORKDIR `/app`).

  > Este comando es útil cuando los archivos se han creado o editado en un sistema Windows o en un entorno que
  > utiliza retornos de carro, y se deben utilizar en un entorno Linux o Unix donde se espera el carácter de nueva
  > línea para indicar el final de una línea. Al eliminar los caracteres de retorno de carro, se asegura que el
  > archivo sea compatible con el sistema en el que se está utilizando.

#### ⚙️ `RUN ./mvnw clean package -DskipTests`

Aquí ocurre la magia. Docker invoca al `Maven Wrapper` (`mvnw`) interno para descargar dependencias y compilar el
`Fat JAR`.

- Este comando se ejecuta en la raíz del directorio de trabajo WORKDIR `/app`.
- **Ventaja:** No necesitas tener Maven instalado en tu PC; el contenedor se encarga de todo.
- **Ubicación:** El JAR resultante se alojará en `/app/target/`.

#### 🔌️`EXPOSE 8001` — instrucción meramente informativa (opcional)

La instrucción `EXPOSE` en un `Dockerfile` tiene como propósito informar que un `contenedor` utiliza un
puerto específico para recibir conexiones. Es una especie de `documentación` dentro del Dockerfile que indica a otros
usuarios qué `puerto expone tu aplicación dentro del contenedor`. En nuestro caso la aplicación de Spring Boot tiene
configurado el puerto `server.port=8001`, eso significa que la aplicación estará escuchando en el puerto `8001` dentro
del contenedor. En este caso, debemos `exponer` ese mismo puerto en el `Dockerfile`. Esto asegura que el puerto `8001`
esté documentado y visible en el contenedor para quien necesite mapearlo o conectarse.
> Al final de este README hablaré un poco más sobre el EXPOSE definido en el Dockerfile y la opción `-P` o
> `--publish-all`.

#### 🚀 `ENTRYPOINT` con ruta interna

A diferencia del `Dockerfile` anterior, donde nosotros renombrábamos manualmente el archivo a `app.jar`
durante la copia, aquí apuntamos directamente a la ruta donde `Maven` lo depositó:
`./target/user-service-0.0.1-SNAPSHOT.jar`.

**¿Por qué el nombre completo?**  
Debido a que el artefacto se construye de forma automatizada al interior del contenedor, debemos referenciar
el nombre exacto que genera el motor de Maven según nuestro `pom.xml`. Esto garantiza que estamos ejecutando
la versión recién compilada dentro del entorno aislado de Docker.

### 💡 Observación de `SonarLint` y Capas de Docker

En esta instrucción, estamos uniendo dos procesos en un solo comando `RUN` mediante el operador `&&`,
siguiendo las recomendaciones de `SonarLint` y las buenas prácticas de `Docker`:

````dockerfile
RUN sed -i -e 's/\r$//' ./mvnw && ./mvnw clean package -DskipTests
````

¿Por qué es una mejor práctica?

1. `Eficiencia de Capas`: Cada instrucción `RUN` genera una nueva capa en la imagen. Al unirlos, creamos una única
   capa de construcción, reduciendo el número de capas intermedias y el tamaño final de la imagen.
2. `Atomicidad`: Nos aseguramos de que el `mvnw` solo se ejecute si el saneamiento de caracteres `(sed)` fue exitoso.
   Si el primer comando falla, el segundo no se ejecuta, evitando imágenes corruptas.

## ⚡ Optimizando Dockerfile 02: Estrategia de Caché de Capas

En el apartado anterior logramos automatizar el build, pero introdujimos una ineficiencia: cualquier cambio mínimo en
una clase Java provocaba que Docker descartara todo lo anterior y volviera a descargar todas las dependencias de Maven.

### 🧠 El Concepto: Invalidez de Capas

Docker construye imágenes por capas. Si una capa cambia (por ejemplo, el código fuente), **todas las capas siguientes
quedan invalidadas** y deben ejecutarse de nuevo.

**El problema anterior:** Teníamos `COPY ./ ./` antes del `RUN`. Como el código cambia frecuentemente, Docker siempre
ejecutaba el `RUN` desde cero.

### 🛠️ La Solución: Separar "Lo Estático" de "Lo Dinámico"

Hemos reestructurado el `Dockerfile` para copiar primero los archivos que definen las dependencias
(`pom.xml` y el `wrapper de Maven`) y luego el código fuente.

````dockerfile
FROM eclipse-temurin:25-jdk-alpine
WORKDIR /app

# 1. Copiamos SOLO lo necesario para las dependencias (Lo que menos cambia)
COPY ./mvnw ./
COPY ./.mvn ./.mvn
COPY ./pom.xml ./

# 2. Descargamos dependencias (Esta capa se cacheará si el pom.xml no cambia)
RUN sed -i -e 's/\r$//' ./mvnw && ./mvnw dependency:go-offline

# 3. Copiamos el código fuente (Lo que más cambia)
COPY ./src ./src

# 4. Empaquetamos (Usando las dependencias ya descargadas en el paso 2)
RUN ./mvnw clean package -DskipTests

EXPOSE 8001
ENTRYPOINT ["java", "-jar", "./target/user-service-0.0.1-SNAPSHOT.jar"]
````

#### 📦 `dependency:go-offline`

Este es el comando clave de Maven. Le indica que descargue todas las librerías necesarias definidas en el
`pom.xml` para que el proyecto pueda compilarse más tarde sin necesidad de conexión a internet.

- `Resultado en Docker`: Si no has añadido una nueva dependencia en tu `pom.xml`, Docker verá que los archivos copiados
  en el paso 1 son idénticos y dirá: `CACHED`. Se saltará la descarga por completo.

#### 📂 `COPY ./src ./src`

Al mover esta instrucción después de la descarga de dependencias, protegemos el proceso pesado de Maven.
Si modificas un `Controller` o un `Service`, Docker solo invalidará a partir de esta línea.

### 💎 Beneficios de esta Estructura

Con esas modificaciones realizadas a nuestro `Dockerfile`, cada vez que cambiemos algo en el código fuente,
las dependencias ya no volverán a descargarse, porque lo que modificamos fue el código fuente y no las dependencias,
por lo tanto, **la velocidad de creación de la imagen será más rápida.**

1. `Velocidad Extrema`: Las construcciones posteriores a un cambio de código ahora tardarán segundos en lugar de
   minutos.
2. `Ahorro de Ancho de Banda`: Dejamos de descargar gigabytes de librerías innecesariamente.
3. `Orden Lógico`: Separamos la infraestructura del proyecto (Maven) de la lógica de negocio (Código fuente).

> `Nota`: Aunque esto mejora drásticamente el tiempo de construcción, el tamaño de la imagen seguirá siendo similar (~
> 494MB) porque seguimos manteniendo el JDK y el código fuente dentro. ¡Pero ya llegaremos a esa parte!

### 📝 Notas de Profundización Técnica

#### Nota 01: El Corazón del Maven Wrapper `(.mvn)`

> En el segundo `COPY` estamos copiando el directorio `.mvn` y su contenido a la imagen de Docker, pero,
> `¿qué es ese directorio?`.
>
> El directorio `.mvn` en una aplicación de `Spring Boot` es un directorio especial que se utiliza para alojar archivos
> relacionados con la construcción y configuración del proyecto. En particular, el directorio `.mvn` **se utiliza para
> personalizar la construcción del proyecto utilizando el mecanismo de "wrapper" de Maven.**
>
> El `Maven Wrapper (o simplemente "wrapper")` **es una forma de garantizar que un proyecto se construya con una versión
> específica de Maven**, independientemente de la versión de Maven instalada en el sistema del desarrollador. Esto puede
> ser útil para garantizar que todos los miembros del equipo utilicen la misma versión de Maven y para simplificar la
> configuración del entorno de construcción.
>
> Dentro del directorio `.mvn`, normalmente encontrarás dos archivos clave:
>
> - `wrapper/ (subdirectorio):` Este subdirectorio contiene los archivos necesarios para el `Maven Wrapper`.
> - `maven-wrapper.properties` especifica la versión de Maven que se utilizará y cómo se descargará si no está presente.
> - `maven-wrapper.jar` es una biblioteca que permite ejecutar Maven sin tenerlo instalado localmente.

#### Nota 02: El Objetivo `dependency:go-offline`

> `RUN ./mvnw dependency:go-offline`:<br>
> - `dependency:go-offline`, objetivo (goal) que resuelve todas las dependencias del proyecto, incluyendo plugins e
    informes y sus dependencias. Después de ejecutar este objetivo, podemos trabajar con seguridad en modo offline.
> - El objetivo `dependency:go-offline` descarga todas las dependencias del proyecto y las almacena en el repositorio
    local de Maven en la imagen de Docker.
> - Esto es útil para garantizar que todas las dependencias estén disponibles sin necesidad de una conexión a Internet
    durante la construcción de la imagen de Docker.
> - Esta instrucción no construye el proyecto ni empaqueta la aplicación Spring Boot.
>
> `Conclusión`: utilizamos esa instrucción, ya que solo deseamos descargar las dependencias y preparar el entorno de
> `Maven` para una construcción posterior.

### 🧪 Pruebas de Optimización 02: Estrategia de Caché de Capas

En este apartado validaremos empíricamente cómo la reestructuración del `Dockerfile` reduce drásticamente los
tiempos de construcción al reutilizar las capas de dependencias.

#### 🧹 1. Limpieza del Contexto (Build Cache)

Antes de medir, debemos asegurar un escenario de "primera ejecución". Docker almacena capas de construcciones previas
que podrían falsear nuestros resultados.

**🔍 Diagnóstico de Almacenamiento**  
Primero, verificamos el estado del almacenamiento en Docker:

````bash
$ docker system df                                        
TYPE            TOTAL     ACTIVE    SIZE      RECLAIMABLE 
Images          15        0         5.825GB   16.62MB (0%)
Containers      0         0         0B        0B          
Local Volumes   0         0         0B        0B          
Build Cache     112       0         1.935GB   1.935GB     
````

Interpretación de resultados:

- `Build Cache`: Representa las capas intermedias (como la descarga de Maven). Si vemos un tamaño alto (ej. `1.935GB`)
  y un porcentaje de `RECLAIMABLE` del `100%`, significa que tenemos caché acumulada de proyectos anteriores
  que podemos borrar.

Para eliminar toda la caché de construcción sin afectar nuestras imágenes finales ni contenedores activos, utilizamos:

> - Resultados intermedios de instrucciones del Dockerfile (RUN, COPY, etc.).

> Solo eliminamos la caché de construcción (build cache), y con `-a` eliminamos toda la caché, no solo la que
> no está en uso.

## 🏗️ Optimizando Dockerfile 03: Multi-Stage Builds & Layer Tools

En el apartado anterior logramos velocidad, pero el tamaño de la imagen seguía siendo un problema `(~531MB)`. Esto se
debe a que la imagen final cargaba con el JDK completo, el código fuente y las herramientas de Maven.

Para resolver esto, utilizaremos `Multi-stage builds` (compilaciones multietapa), una técnica que permite usar imágenes
pesadas para construir y entregar una imagen final extremadamente ligera.

> Para tener mayor detalle sobre la `creación de una imagen de docker eficiente` visitar mi repositorio
> https://github.com/magadiflo/spring-microservices-in-action-2021/blob/main/04.welcome-to-docker.md

### 🧩 [¿Qué es Multi-stage Build?](https://docs.docker.com/build/building/multi-stage/)

Permite utilizar múltiples instrucciones `FROM` en un solo `Dockerfile`. Cada `FROM` inicia una nueva etapa y nos
permite copiar artefactos de forma selectiva de una etapa a otra, descartando todo lo innecesario
(código fuente, caché de Maven, etc.) en la imagen final.

### 🛠️ El Nuevo Dockerfile: Triple Etapa

Hemos evolucionado el archivo a un flujo de tres estaciones de trabajo:

````dockerfile
# --- STAGE 1: Descarga y Empaquetado (Dependencies) ---
FROM eclipse-temurin:25-jdk-alpine AS dependencies
WORKDIR /app
COPY ./mvnw ./
COPY ./.mvn ./.mvn
COPY ./pom.xml ./
RUN sed -i -e 's/\r$//' ./mvnw && ./mvnw dependency:go-offline
COPY ./src ./src
RUN ./mvnw clean package -DskipTests

# --- STAGE 2: Extracción de Capas (Builder) ---
FROM eclipse-temurin:25-jre-alpine AS builder
WORKDIR /app
COPY --from=dependencies /app/target/*.jar ./app.jar
RUN java -Djarmode=layertools -jar app.jar extract

# --- STAGE 3: Imagen Final de Ejecución (Runner) ---
FROM eclipse-temurin:25-jre-alpine AS runner
WORKDIR /app
COPY --from=builder /app/dependencies ./
COPY --from=builder /app/spring-boot-loader ./
COPY --from=builder /app/snapshot-dependencies ./
COPY --from=builder /app/application ./
EXPOSE 8001
ENTRYPOINT ["java", "org.springframework.boot.loader.launch.JarLauncher"]
````

#### 1️⃣ Etapa: `dependencies` (El Constructor)

- Usa el JDK (Java Development Kit) porque necesita compilar.
- Su único objetivo es generar el archivo `.jar`. Al terminar esta etapa, todo el código fuente y Maven se quedan aquí y
  no pasan a la siguiente.

#### 2️⃣ Etapa: `builder` (El Extractor)

- Aquí ocurre algo avanzado: usamos `-Djarmode=layertools`.
- Concepto Clave: Spring Boot permite "explotar" el JAR en 4 capas lógicas (`dependencies`, `spring-boot-loader`,
  `snapshot-dependencies` y `application`).
- Esto es vital para que, en futuros despliegues, si solo cambia tu código, Docker solo actualice la capa
  `application` (que pesa KB) y no las librerías (que pesan MB).

#### 3️⃣ Etapa: `runner` (La Imagen Final)

- Usa el `JRE` (Java Runtime Environment), que es mucho más pequeño y seguro porque no contiene compiladores.
- `Copiado Selectivo`: Solo traemos las carpetas extraídas en la etapa anterior.
- `JarLauncher`: Ya no ejecutamos `java -jar`. Usamos el lanzador nativo de Spring para arrancar la aplicación de forma
  más eficiente desde las capas extraídas.

  librerías de terceros (que suelen ser el 90% del peso) no han sufrido cambios, ahorrando tiempo y ancho de banda.
- `Selectividad`: Solo se ejecutan los pasos que dependen del código fuente (`COPY ./src` y el empaquetado final).
- `Eficiencia de Capas`: Gracias a `layertools`, Docker solo tuvo que re-generar la capa de `application`. Las capas de
  `dependencies` y `spring-boot-loader` se mantuvieron intactas en la imagen final.

## 📑 Instrucción `EXPOSE` del Dockerfile

En Docker, el comando `EXPOSE` se utiliza para indicar qué puertos serán expuestos por el contenedor cuando se ejecute.
Sin embargo, **es importante destacar que `EXPOSE` no publica realmente los puertos. Lo que hace es documentar los
puertos en los que una aplicación dentro del contenedor escuchará conexiones.**

Por ejemplo, si tu `aplicación Spring Boot escucha en el puerto 8001 (configurado con el server.port)` tu archivo
`Dockerfile` podría tener una línea como esta:

````dockerfile
EXPOSE 8001
````

Esto significa que la aplicación dentro del contenedor estará escuchando en el puerto 8001 para las conexiones
entrantes. Indica que el contenedor expone el puerto 8001 para ser accesible desde otros contenedores en la misma red
de Docker. Pero para acceder a ese puerto desde fuera del contenedor, deberías mapearlo al puerto de tu host cuando
ejecutes el contenedor:

````bash
$ docker container run -p <HOST_PORT>:<CONTAINER_PORT> <nombre_de_la_imagen>
````

- `HOST_PORT`: Es el puerto de tu pc local. Puede ser igual al puerto del contenedor o diferente (ej. 8080).
- `CONTAINER_PORT`: Debe coincidir estrictamente con el puerto definido en tu `application.properties` de Spring Boot
  (ej. `server.port=8001`) y con la instrucción `EXPOSE` del `Dockerfile`.

En resumen, la instrucción `EXPOSE` de tu `Dockerfile` debería usar el puerto interno de tu contenedor (usualmente el
puerto en el que tu aplicación Spring Boot está escuchando). Luego, al ejecutar el contenedor, especificas cómo deseas
que los puertos del host se relacionen con los puertos expuestos del contenedor.

Más adelante parametrizaremos los valores de nuestro `Dockerfile`, así que cuando lleguemos a la instrucción del
`EXPOSE` lo crearemos de la siguiente manera.

````dockerfile
EXPOSE ${CONTAINER_PORT}
````

# 📘 Libro: Aprender Docker

## Publicación Automática: La Opción -P (--publish-all)

Recordemos que cuando creamos un `Dockerfile` podemos definir una instrucción llamada `EXPOSE`, que según la
documentación oficial de Docker, es usado simplemente como documentación, pero si creamos un contenedor con la
opción `-P` (mayúscula) podemos hacer uso de ella.

Con la opción `-P` (en mayúscula) publicamos todos los puertos que se hayan expuesto en la imagen que se ha utilizado
para crear el contenedor, en puertos aleatorios del `host`.

En este caso, no tenemos que indicar el puerto local, sino que **será seleccionado aleatoriamente** entre los puertos
que estén libres. El puerto del contenedor con el que se hace la redirección estará definido en el `Dockerfile`
con el que se ha creado la imagen del contenedor (usando la instrucción `EXPOSE`). Tenga en cuenta que, en el archivo
`Dockerfile`, se pueden exponer varios puertos a la vez.

**🚨 Importante**

> El siguiente ejemplo no es parte del proyecto, simplemente es para ver el funcionamiento de la opción
> `-P (mayúscula)` y la instrucción `EXPOSE`.

### 🧪 Ejemplo Práctico: Aplicación products

`Supongamos` que mi aplicación de Spring Boot se llama `products` y está usando el puerto `8085`:

````yaml
server:
  port: 8085

spring:
  application:
    name: products
````

Eso significa que el puerto que usará internamente el contenedor debe ser el `8085`.

Una vez construido el `jar` generamos la imagen de la aplicación utilizando el siguiente `Dockerfile`:

````dockerfile
FROM eclipse-temurin:25-jdk-alpine AS builder
WORKDIR /app
COPY ./target/*.jar ./app.jar
ARG CONTAINER_PORT=8085
EXPOSE ${CONTAINER_PORT}
CMD ["java", "-jar", "app.jar"]
````

Este `Dockerfile` define el puerto `CONTAINER_PORT`, que representa el puerto en el cual se ejecutará nuestra aplicación
Spring Boot dentro del contenedor. En este caso, el valor por defecto es `8085`.

A continuación, construimos una imagen a partir del `Dockerfile` anterior.

````bash
$ docker image build -t products .
````

Luego, creamos un contenedor utilizando la imagen recién generada.

````bash
$ docker container run -d -P --name c_products products
````

Es importante notar que estamos utilizando la opción `-P (mayúscula)`. Esta opción le indica a `Docker` que asigne
automáticamente un puerto aleatorio disponible en el host y lo redirija al puerto `8085` expuesto por el contenedor.

Listamos los contenedores para ver el que acabamos de crear:

````bash
$ docker container ls -a

b787f2484e07   products               "/__cacert_entrypoin…"   6 seconds ago   Up 4 seconds       0.0.0.0:32775->8085/tcp            c_products
````

Comprobamos que **el puerto aleatorio que se ha seleccionado para nuestro host Docker (pc local)** es el puerto `32775`
que está siendo enlazado al puerto `8085` dentro del contenedor.

Comprobamos que podemos acceder al contenido del contenedor usando la terminal con el comando curl:

